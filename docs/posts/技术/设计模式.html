设计模式

🕒 Published at:

设计模式的六大原则 ​

对接口编程而不是对实现编程。 优先使用对象组合而不是继承。

设计模式的使用（solid）

  • 单一职责(Single Responsibility)
  • 开闭原则（Open Close Principle）:对扩展开放，对修改关闭
  • 里氏代换原则（Liskov Substitution Principle）：继承复用,子类替代父类出现。
  • 迪米特法则（Law Of Demeter Principle）：一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。
  • 接口隔离原则（Interface Segregation Principle）：使用多个隔离的接口，比使用单个接口要好
  • 依赖倒转原则（Dependence Inversion Principle）：针对接口编程，依赖于抽象而不依赖于具体。
  • 合成复用原则（Composite Reuse Principle）：尽量使用合成/聚合的方式，而不是使用继承。

创建型 ​

工厂模式（Factory Pattern） ​

作用：创建复杂对象 应用：Hibernate 换数据库只需换方言和驱动 优缺点：拓展方便、类多

抽象工厂模式（Abstract Factory Pattern） ​

作用：创建一系列复杂对象 应用：1、QQ 换皮肤，一整套一起换。 2、生成不同操作系统的程序。 优缺点：拓展困难

单例模式（Singleton Pattern） ​

作用：创建一系列复杂对象 应用：饱汉、饿汉、DLC

## DLC例子
public class T {
	private static volatile T t;
	private T() {}
	public static T getT() {
		//DoubleChecking双重检测--->避免不必要的同步
		if(null!=t) {
			return t;
		}
		synchronized (T.class) {
			if(null==t) {
				t=new T();
				//1、开辟空间 2、初始化对象信息 3、返回对象的地址给引用
				//这里可能会发生指令重排，导致返回给的对象是一个空对象，故volatile
			}
		}
		
		return t;
	}
}

结构型 ​

原型模式（Prototype Pattern） ​

作用：创建重复的对象，同时又能保证性能。 实现： Cloneable接口

[重]适配器模式（Adapter Pattern） ​

作用：结合了两个独立接口的功能 实现：适配器继承新接口，依赖旧接口。(MediaAdapter.java)

桥接模式（Bridge Pattern） ​

作用：抽象化与实现化解耦，使得二者可以独立变化 实现：抽象类依赖实现类。

过滤器模式（Filter、Criteria Pattern） ​

作用：过滤 实现：就一个循环过滤...

组合模式（Composite Pattern） ​

作用：层次结构 实现：树。

[重]装饰器模式（Decorator Pattern） ​

作用：向一个现有的对象添加新的功能，同时又不改变其结构 实现：抽象类implements Shape 和 protected Shape decoratedShape;

外观模式（Facade Pattern） ​

作用：隐藏系统的复杂性 实现：外观类持有对象并封装细节。

享元模式（Flyweight Pattern） ​

作用：减少创建对象的数量 实现：hashmap缓存等

[重]代理模式（Proxy Pattern） ​

作用：减少创建对象的数量 实现：代理类实现一样的接口并持有被代理对象。（与装饰器模式区别：不关心被代理对象状态。）

行为型 ​

责任链模式（Chain of Responsibility Pattern） ​

作用：如果一个对象不能处理该请求，那么它会把相同的请求传给下一个接收者。降低耦合，灵活性，拓展性。 实现：Handler 里面聚合它自己

[重]命令模式（Command Pattern） ​

作用：调用对象寻找可以处理该命令的合适的对象，并把该命令传给相应的对象，该对象执行命令

[重]观察者模式（Observer Pattern） ​

作用：当一个对象被修改时，则会自动通知依赖它的对象 实现：对象subject持有多个观察者，改变时触发更新。

[重]状态模式（State Pattern） ​

作用：类的行为是基于它的状态改变的，条件、分支语句的代替者。 实现：context持有state接口

空对象模式（Null Object Pattern） ​

空对象代替null,该空对象类将无缝地使用在需要检查空值的地方

[重]策略模式（Strategy Pattern） ​

作用：类的行为或其算法可以在运行时更改，避免使用多重条件判断，扩展性良好。 实现：context持有策略接口，封装方法。策略较多时看：混合模式，

模板模式（Template Pattern） ​

作用：封装不变部分，扩展可变部分，提取公共代码，便于维护 实现：抽象父类控制行为，子类负责实现

访问者模式（Visitor Pattern） ​

作用：不同的访问者类，不同的执行算法。 实现：被观察领域的接口方法传入访问者接口。
